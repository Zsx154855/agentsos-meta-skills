## 心智模型（5 个核心）

### 1. 硬件民主化（Hardware Democratization）

**一句话**：让 AI 工具在绝大多数人能拥有的硬件上运行。兼容性 > 性能峰值，普惠是工程优先级的顶层。

**来源**：
- SD WebUI 持续优化 4GB VRAM 支持，并有 2GB VRAM 的运行报告
- `--medvram`、`--lowvram`、`--opt-split-attention` 等优化选项是内置的，不是 hack
- 大量用户反馈围绕低端硬件使用。这是 GitHub Issues 的核心关切
- 去中心化的 AI 生成意味着 AI 应该在任何电脑上运行，不依赖云服务

**应用方式**：
- 思考你的目标用户的"最小公共硬件"是什么。支持它，而不是把它排除在外
- 提供从"兼容模式"到"全性能模式"的渐进配置，而不是一刀切
- 在所有优化路径上测试，不仅仅是你的开发机
- 当性能优化与兼容性冲突时，默认选兼容性

**局限**：2GB/4GB VRAM 的限制可能阻止采用最新模型架构；优化低端硬件需要持续的工程投入；某些功能（如超大分辨率生成）在低 VRAM 下难以实现。

---

### 2. 实验迭代（Iterative Experimentation as First-Class）

**一句话**：AI 生成不是一次性的输出，而是探索性的对话。工具必须支持"尝试→对比→调整→重试"的循环，并且要把它做成一等公民。

**来源**：
- X/Y/Z plot 脚本——批量对比不同参数、不同模型、不同 prompt 的效果网格
- Loopback 功能——生成的输出直接作为下一步的输入，实现迭代式 generation
- Seed resizing 和 variation 种子——在不完全改变图像的情况下控制变化幅度
- Interrupt processing——用户可以随时打断生成，不必等完成
- img2img + denoising strength——从噪声控制到原图保留的连续谱

**应用方式**：
- 任何 AI 工具都应该有"对比模式"——让用户同时看到多个输出，而不是一个
- 打断和恢复应该是基本操作。永远不要强迫用户等待
- 参数空间应该是可搜索的——X/Y/Z plot 本质上是参数空间扫描
- 提供连续谱控制（如 denoising strength 0-1），而不是离散选项
- 前后对比应该被系统性地支持

**局限**：实验模式（grid 生成）消耗更多计算资源；某些用户只需要简单的一次性输出；过多的参数控制可能造成新用户困惑。

---

### 3. 插件即架构（Extensions as Architecture）

**一句话**：每个主要能力都是可插拔的。扩展不是附加功能——它是系统架构的第一公民。核心仓库只做一件事：让扩展能高效工作。

**来源**：
- 上采样（Upscaling）、面部修复（Face restoration）、提示权重（prompt weighting）、模型合并（checkpoint merging）——全部作为插件或脚本实现
- Extensions 标签页是 webui 的核心导航项
- 内置扩展仓库（Extension index）——社区扩展的发现和安装直接在 UI 内完成
- 扩展 API 支持：自定义脚本（custom scripts）、UI 注入（gradio blocks）、后处理（postprocessing）

**应用方式**：
- 每当你考虑添加新功能，先问：这应该是一个插件还是核心？
- 核心仓库只做"让插件高效工作"的基础设施：UI 框架、参数传递、模型加载、图像 IO
- 插件接口必须稳定、文档化、版本化
- 提供示例扩展（Example extensions）：让社区知道怎么写扩展
- 扩展的安装和更新应该和核心仓库一样简单

**局限**：插件架构增加了维护负担——每次 API 变动都需要更新大量扩展；插件质量参差不齐；极端情况下的安全风险。

---

### 4. 过程即产物（Process as Artifact）

**一句话**：生成的每一帧都应该可以完整复现。参数、种子、模型、prompt、时间戳——全部嵌入在 PNG chunk 中。拖回一张图片就能恢复完整生成环境。

**来源**：
- PNG info 页面——从 PNG chunk 读取生成参数
- 所有参数保存在 PNG 元数据中（prompt、negative prompt、seed、CFG、model hash、VAE 等）
- 批量处理——每个图像文件的参数结构一致
- 种子参数化——用种子追踪和复现精确输出
- "Read generation parameters from prompt"——从任何文本恢复参数

**应用方式**：
- 每个输出文件应该包含足够的信息以完全复现它
- 使用标准化的元数据格式（PNG chunks、EXIF、JSON sidecar）
- 提供"参数恢复"功能：将文件拖回 UI 就能复现生成结果
- 种子管理：支持精确种子、随机种子、种子变体（variation）
- 批处理的元数据应保持一致结构

**局限**：元数据增加文件体积；某些图像处理管道会剥离元数据；参数版本化（不同软件版本的参数兼容性）需要持续管理。

---

### 5. 提示词即语言（Prompt as Composable Language）

**一句话**：提示词不是简单的文本输入——它是一个可组合、可运算的领域特定语言。用语法表达注意力、权重、交叉注意力控制和概念混合，让文字成为画笔。

**来源**：
- 注意力语法：`(word:1.2)` 增加权重，`[word:0.8]` 降低，`(word:1.2:0.5)` 控制步骤权重
- 提示编辑：`[from:to:step]` 在生成过程中切换 prompt
- AND 分隔：`cat AND dog` 混合多个概念
- 交替提示：`[cat|dog|bird]` 按步数交替概念
- 负面提示词（Negative prompt）——告诉模型不要生成什么
- Prompt weighting 和 SD 注意力控制（cross-attention control）是社区扩展的标准化
- 高级功能：循环提示、时间线提示、区域提示、动态提示

**应用方式**：
- 把文本输入视为 DSL 的设计空间——语法扩展应该像语言设计一样认真
- 权重控制是核心需求：用户需要"这个更重要/更不重要"的表达
- 时间序列控制（何时出现何时消失）是生成式 AI 独有的表达能力
- 组合（AND、混合）打开了概念编辑的大门
- 负面表达（negative prompt）是创造性控制的关键机制

**局限**：DSL 越丰富，学习曲线越陡；语法更新可能破坏已有的 prompt 文件；高级语法与不同模型的兼容性有差异；新用户可能被语法吓到而非赋能。

## 决策启发式（7 条）

1. **如果一个功能可以通过插件实现，就不要放进核心仓库。** 核心的职责是让插件能更好地工作。

2. **支持最慢的显卡意味着你关心最多的人。** 硬件兼容性是功能，不是妥协。

3. **用户每次生成都应该得到一个"记录日"** ——参数、种子、模型、版本。可复现性是 AI 工具的基本人权。

4. **提供一个按钮让用户做 X/Y/Z 对比。** 在探索性工作中，比较和迭代比单次输出重要得多。

5. **让你的工具在没有网络的情况下也能工作。** 去中心化的 AI 是长期价值。

6. **UI 布局应该是用户可配置的。** 专家用户知道什么控件离他们最近。

7. **在显示成功之前允许用户打断。** 不是每个人都需要 50 步的完整扩散过程。

## 内在张力

1. **功能丰富 vs 新用户入门**：几百个参数、几十个扩展、无数脚本 vs 用户只想生成第一张图。解决方案：默认值够好 + 社区教程。

2. **插件生态 vs 核心质量**：每个新扩展带来功能也带来问题（冲突、性能、安全）。核心团队不可能审查每一个扩展。

3. **低端硬件 vs 最新模型**：新模型通常需要更多 VRAM。支持旧硬件意味着错过新模型时代的窗口。

4. **用户控制 vs 简单体验**：给用户 50 个滑块是赋能还是困惑？AUTOMATIC1111 选择了"先全给，让社区教怎么简化"。

5. **开源免费 vs 维护成本**：164k stars 带来了巨大的 Issue 和 PR 负担。项目规模的维护需要超越个人能力。

## 关键架构决策

### 为什么是 Gradio 而不是 React/Vue？

Gradio 让 Python 后端的开发者不需要前端专家就能快速构建交互界面。核心哲学：**AI 研究者构建 AI 工具应该优先于前端架构的优雅**。

### 为什么是单页而不是多页面？

所有功能在同一个页面。参数、生成、历史、设置都在一个滚动视图里。目的是**让所有操作在可见范围内完成**，不打断生成流程。

### 为什么 Extensions 是核心 Tab 而不是单独的安装器？

把扩展管理和核心功能放在同一级，说明**扩展不是后加的装饰——它是系统架构的有机组成部分**。

### 为什么是 PNG Info 而不是单独的参数文件？

PNG 是人类和机器都能读的标准格式。把参数嵌入图像本身，参数和输出永远不会分离。
